# Waterboyz User Guides — Design Spec

**Date:** 2026-04-20
**Author:** James Davies (with Claude)
**Status:** Approved, ready for implementation planning

## Goal

Produce professional, branded user-facing documentation for two Waterboyz portal audiences:

1. **SWAT Leadership** — admins who review nominations, manage volunteers, run reports, and coordinate events
2. **Team Captains** (a.k.a. Project Captains) — volunteers assigned to lead individual SWAT projects

Each guide includes a **Quick Start** (first few pages) followed by **comprehensive** coverage of every feature currently available in the portal. Screenshots are captured via Playwright against production data.

Both guides are delivered in two forms:

- **Standalone HTML** in `docs/user-guides/dist/` — self-contained, shareable, print/PDF friendly
- **In-portal React pages** at `/help/swat-leaders` and `/help/team-captains` — behind authentication, bundled into the portal so screenshots are not publicly reachable

## Non-Goals

- No new CMS or authoring UI — content lives as structured JS files in the repo
- No CI/Playwright integration — screenshot capture is a manual developer workflow; generated React components are committed so CI does not need to run Playwright
- No auto-publishing to external hosts (e.g. Notion, S3) — standalone HTML can be shared manually
- No versioning / history for the docs — they reflect the current state of the portal and are regenerated on each build

## Architecture

### Directory layout

```
docs/user-guides/
  build.js                  # shared build pipeline — generates both outputs
  brand.js                  # shared brand tokens (logo URL, colors, fonts)
  capture.ts                # Playwright screenshot capture script
  .env.example              # template for local credentials
  content/
    swat-leaders/
      00-quick-start.js
      01-home.js
      02-nominations.js
      03-volunteer-management.js
      04-reports.js
      05-events.js
      06-settings.js
    team-captains/
      00-quick-start.js
      01-my-projects.js
      02-project-details.js
      03-volunteers.js
      04-tasks.js
      05-expenses.js          # Coming Soon — placeholder section
      06-photos-notes.js
      07-requesting-help.js
  screenshots/              # PNGs captured by capture.ts (committed)
  dist/                     # generated standalone HTML (committed)

portal/src/pages/help/
  HelpLayout.jsx            # sidebar TOC + breadcrumbs, inside authenticated shell
  SwatLeadersGuide.jsx      # route component
  TeamCaptainsGuide.jsx     # route component

    swat-leaders.jsx
    team-captains.jsx
  assets/                   # screenshots copied from docs/user-guides/screenshots/
```

### Data flow

```
content/*.js  ──┐
                ├─→  build.js  ──┬─→  dist/*.html             (standalone)
screenshots/  ──┘                └─→  portal/src/pages/help/  (React bundle)
```

Running `node docs/user-guides/build.js` regenerates **both** outputs from the current content + screenshots. Commit the generated files so CI does not need Playwright.

### Authentication & hosting

- Standalone HTML lives in `docs/user-guides/dist/` — not served by the portal
- In-portal pages register routes `/help/swat-leaders` and `/help/team-captains` inside the existing authenticated router (same guard that wraps `/admin/*`)
- Screenshots are imported as Vite modules from `portal/src/pages/help/assets/` so they are bundled into the authenticated JS asset bundle — not publicly reachable via a static URL
- Role-aware hamburger menu entry (**Help**): SWAT Leadership and Space Admin see both guides; other authenticated users see only the Team Captain guide

## Content Structure

### Quick Start (first section of each guide)

Two short pages that orient a new user in under 5 minutes:

1. **What you do in this role** — 3–4 bullet overview
2. **Your first 5 minutes** — log in → find X → do Y, with one annotated screenshot per step
3. **Where to go for common tasks** — small table linking into the comprehensive sections

### Comprehensive section template

Each section uses the same shape:

1. **Purpose** — 1–2 sentences: what this page/feature is for
2. **How to get there** — navigation path with screenshot of the entry point
3. **Walkthrough** — numbered steps; one screenshot per step or per group of related steps
4. **Field reference** — where relevant (e.g. project status meanings, volunteer fields)
5. **Tips & gotchas** — callout box with domain notes (e.g. "Status 'Removed' is soft-delete")

### Content file format

```js
// content/swat-leaders/03-volunteer-management.js
export default {
  slug: 'volunteer-management',
  title: 'Volunteer Management',
  purpose: 'Search, filter, and manage your entire volunteer directory...',
  howToGetThere: {
    text: 'From the hamburger menu, open Admin → Volunteer Management.',
    screenshot: 'swat-leaders/vm-nav.png',
  },
  steps: [
    {
      heading: 'Filtering by skill',
      body: 'Click the Skills column header...',
      screenshot: 'swat-leaders/vm-skill-filter.png',
    },
  ],
  fields: [
    { name: 'Service Area', description: 'Counties the volunteer is willing to travel to.' },
  ],
  tips: [
    "The 'Removed' status on a project assignment is soft-delete — the record is preserved for history.",
  ],
  comingSoon: false,
};
```

`build.js` walks each role's content folder in filename order, resolves screenshot paths, and emits both outputs.

## Playwright Capture (`docs/user-guides/capture.ts`)

- Standalone Playwright script (not part of `frontend-testing/`) to keep test output clean
- Reads credentials from `docs/user-guides/.env` (template in `.env.example`):
  ```
  PW_BASE_URL=https://waterboyz.kinops.io
  PW_LEADER_USERNAME=...
  PW_LEADER_PASSWORD=...
  PW_CAPTAIN_USERNAME=...
  PW_CAPTAIN_PASSWORD=...
  SAMPLE_PROJECT_ID=...
  SAMPLE_VOLUNTEER_ID=...
  SAMPLE_EVENT_ID=...
  SAMPLE_NOMINATION_ID=...
  SAMPLE_FAMILY_ID=...
  ```
- Viewport **1440×900** for desktop captures; **390×844** (iPhone 14) for mobile-only views (home nav, responsive Volunteer Management card list)
- Each screenshot is a named recipe function; the script is idempotent — reruns overwrite existing PNGs
- Uses `page.locator(...).screenshot({ path })` where possible to crop to the relevant region
- Honors nav state: if a nav entry is disabled, the script records that fact so `build.js` can emit a "Coming Soon" section without walkthrough screenshots

### PII handling — hybrid (decision C)

- **Synthetic where easy:** a small set of sample records tagged for screenshot use
  - 1 sample Family (with 2 family members)
  - 1 sample Volunteer
  - 1 sample SWAT Project (assigned the sample volunteer, linked to the sample family)
  - 1 sample Event (with the sample volunteer signed up)
  - 1 sample Nomination (for the approval walkthrough)
- **Blur where not** (aggregates / dashboards where we can't control the data): inject a `.pw-redact` class before screenshot:
  ```css
  .pw-redact {
    filter: blur(6px);
    color: transparent !important;
    text-shadow: 0 0 8px rgba(0,0,0,0.6);
  }
  ```
  Applied via a known list of selectors per page (volunteer name cells in aggregated lists, family names in reports, etc.)

The implementation plan will list the exact sample records needed so the user can create them before capture runs.

## Build Pipeline (`docs/user-guides/build.js`)

### Commands

- `node build.js` — regenerate both `dist/` and `portal/src/pages/help/generated/`
- `node build.js --capture` — run Playwright capture first, then build
- `node build.js --preview swat-leaders` — open standalone HTML in the default browser
- `node build.js --list` — show available guides and sections

Root-level convenience script: `yarn help:build` runs `node docs/user-guides/build.js`.

### What it generates

**Standalone HTML** (one file per guide in `dist/`):
- Inline CSS (no Tailwind) — self-contained
- Branded header with Waterboyz logo
- Sidebar TOC (sticky)
- `@media print` rules that hide the sidebar, force clean page breaks, and keep screenshots from splitting
- Role badge in header

**React components** (one `.jsx` per guide in `portal/src/pages/help/generated/`):
- Imports each screenshot as a Vite module so it gets bundled
- Exports a default component with all sections, TOC, and tips
- Consumed by `SwatLeadersGuide.jsx` and `TeamCaptainsGuide.jsx` which wrap it in `HelpLayout.jsx`

Both outputs share brand tokens from `brand.js` (colors, fonts, logo URL).

## Portal Integration

### Routes

Register inside the authenticated router (wherever `PrivateRoute` wraps `/admin/*`):

```jsx
<Route path="help/swat-leaders" element={<SwatLeadersGuide />} />
<Route path="help/team-captains" element={<TeamCaptainsGuide />} />
```

### HelpLayout component

- **Left:** sticky sidebar with section TOC (collapses into dropdown on mobile)
- **Main:** content column, max-width ~800px, comfortable line-height
- **Header:** Waterboyz logo + guide title + role badge
- **Footer:** Previous / Next links between sections

Uses portal Tailwind/DaisyUI theme so guides feel native inside the app.

### Hamburger menu

Add **Help** item (Tabler `IconBook`). Role logic:

- SWAT Leadership / Space Admin → submenu with both guides
- Other logged-in users → direct link to Team Captain guide

## Acceptance Criteria

- [ ] `node docs/user-guides/build.js` produces `dist/swat-leaders.html`, `dist/team-captains.html`, and regenerated React components in `portal/src/pages/help/generated/`
- [ ] `yarn help:capture` (or equivalent) runs Playwright against production with the two role accounts, saves PNGs to `screenshots/`, applies blur to designated selectors, and uses sample record IDs from `.env`
- [ ] `/help/swat-leaders` and `/help/team-captains` render inside the authenticated portal with full TOC, all section screenshots, tips callouts, and Coming Soon badge on Expenses
- [ ] Screenshots are NOT reachable via any public URL (Vite-bundled, not in `portal/public/`)
- [ ] Hamburger menu has a role-aware **Help** entry
- [ ] Standalone HTML in `dist/` is self-contained (no external Tailwind), print-friendly, and shareable as a single file
- [ ] Expenses section in Team Captain guide renders a "Coming Soon" callout — no broken walkthrough
